# Ролевая модель системы рейтинга

## Обзор

Система поддерживает роли пользователей админ-панели:

- **ADMIN** — администратор с полными правами
- **MANAGER** — организатор турниров
- **LICENSE_MANAGER** — менеджер лицензий
- **PRESIDIUM_MEMBER** — член президиума (признание результатов турниров)

Пользователю можно назначить **несколько ролей одновременно**. Итоговые права — это объединение прав всех назначенных ролей.

Подробное описание для сотрудников также доступно в админке: **Справка → Роли в админ-панели** (`/admin/help/user-roles`).

## Роли и права доступа

### ADMIN (Администратор)

Полный доступ ко всем функциям системы:

**Турниры**

- Загрузка турниров (Excel, Google Sheets)
- Просмотр всех турниров
- Признание результатов турниров для учёта в рейтинге
- Удаление турниров
- Удаление результатов турниров
- Пересчёт очков

**Игроки (рейтинг)**

- Просмотр, создание, редактирование, удаление игроков

**Лицензионные игроки**

- Просмотр, создание, редактирование, загрузка из Excel, удаление

**Пользователи**

- Просмотр всех пользователей
- Создание пользователей с любыми ролями (включая комбинации ролей)
- Редактирование и удаление любых пользователей (кроме удаления собственной учётной записи)

**Настройки**

- Изменение настроек рейтинга, количества лучших результатов

**Панель управления и аудит**

- Статистика, быстрые действия, логи аудита

### MANAGER (Организатор турнира)

Доступ к работе с рейтингом и турнирами, без администрирования системы:

**Турниры**

- Загрузка, просмотр, редактирование; без удаления турниров и результатов

**Игроки (рейтинг)**

- Просмотр, создание, редактирование; удаление — по правилам бэкенда (нет участия в турнирах)

**Ограничения**

- Нет панели управления, настроек, логов аудита
- Нет раздела «Пользователи»
- Нет раздела «Лицензии» (лицензионные игроки в админке)

### LICENSE_MANAGER (Менеджер лицензий)

Работа со справочником игроков рейтинга и лицензионной базой без доступа к турнирам и администрированию системы.

**Доступные разделы UI**

- Игроки
- Лицензии
- Справка

**Игроки (рейтинг)**

- Просмотр, создание, редактирование и удаление — по тем же правилам API, что у ADMIN/MANAGER (включая ограничения на удаление при участии в турнирах)

**Лицензионные игроки**

- Создание, редактирование, загрузка из Excel и удаление — как у администратора

**Ограничения**

- Нет доступа к турнирам, панели управления, разделу «Пользователи», настройкам и аудиту

### PRESIDIUM_MEMBER (Член президиума)

Роль для валидации (признания) результатов завершённых турниров.

**Доступные разделы UI**

- Турниры
- Справка

**Турниры**

- Просмотр списка турниров и карточки турнира
- Признание результатов завершённого турнира для учёта в рейтинге

**Ограничения**

- Нет загрузки/редактирования/удаления турниров
- Нет доступа к игрокам, лицензиям, пользователям, настройкам и аудиту

## Мульти-роли

- Роли хранятся как массив в `users.roles` (JSON), а `users.role` сохраняется для обратной совместимости.
- JWT содержит `roles` (и `role` для старых сценариев).
- Проверки прав (`requireRole(...)`) работают по правилу: доступ разрешён, если у пользователя есть **хотя бы одна** из требуемых ролей.
- Примеры:
  - `MANAGER + PRESIDIUM_MEMBER` — может вести турнир и отдельно признавать результаты.
  - `LICENSE_MANAGER + PRESIDIUM_MEMBER` — работает с лицензиями и может признавать турниры, но не редактирует турниры.

## Безопасность

### Пароли

- Минимальная длина: 6 символов
- Хешируются с использованием bcrypt (salt rounds: 10)
- Хранятся в БД в зашифрованном виде

### JWT токены

- Срок действия: 7 дней
- Содержат: userId, username, role, roles
- Автоматически проверяются на каждом запросе

## Troubleshooting

### Не вижу раздел «Пользователи»

Раздел доступен только **ADMIN**.

### Не вижу «Панель управления»

Панель управления только у **ADMIN**.

### Ошибка 403 при работе с пользователями

Управление пользователями доступно только **ADMIN**. У **MANAGER** и **LICENSE_MANAGER** этого раздела нет.

### Не могу удалить турнир / результат

Удаление турниров и отдельных результатов — только **ADMIN**.

### Почему турнир не попал в рейтинг после загрузки результатов

После загрузки/замены результатов завершённого турнира они требуют признания.
Попросите пользователя с ролью **PRESIDIUM_MEMBER** или **ADMIN** выполнить действие «Признать результаты».

## Миграция существующих пользователей

Если у вас есть существующие пользователи в таблице `admins`:

1. Миграция автоматически перенесет их в таблицу `users`
2. Всем будет присвоена роль ADMIN
3. Поле `name` будет заполнено значением из `username`
4. Рекомендуется обновить имена пользователей через UI или API

```sql
-- Обновить имя пользователя
UPDATE users SET name = 'Иванов Иван' WHERE username = 'admin';
```

Расширение enum роли в MySQL и поддержка мульти-ролей выполняются миграциями:

- `021_add_license_manager_role.ts`
- `022_add_presidium_and_tournament_results_validation.ts`
- `023_add_user_roles_json.ts`

## Разработка

### Добавление новых ролей

1. Обновите enum в `backend/src/types/index.ts` и зеркально во `frontend/src/types/index.ts`
2. Обновите enum в базе данных (`users.role`, `audit_logs.user_role`) и обработку `users.roles`
3. Обновите middleware (`requireTournamentStaff`, `requirePlayersSectionAccess`, `requireLicensedPlayersEditor`, `requireAdmin` и т.д.) и маршруты в `backend/src/routes/admin.ts`
4. Обновите навигацию и маршруты с `AdminRoleRoute` во `frontend/src/App.tsx` и `AdminLayout.tsx`
5. Дополните `ROLES.md` и статью справки при необходимости
